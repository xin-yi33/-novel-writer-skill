# 长篇写作模式

> 对应 SKILL.md I 节。适用于持续创作超过 5 章的长篇小说，通过内存文件系统解决 AI 上下文窗口限制，实现跨 session 续写。

---

## 为什么需要长篇写作模式？

AI 的上下文窗口有限（通常几千到几万token），无法一次性容纳整部小说的全部内容。长篇写作模式通过一套**内存文件系统**来解决这个问题：

- 只保留当前写作必需的最少信息（核心设定 + 摘要 + 最近2章正文）
- 每写完一章自动更新所有内存文件
- 即使关闭对话再重新开始，也能从断点精确续写
- 避免长篇创作中常见的"写后面忘前面"问题

---

## 何时使用长篇写作模式 vs 常规模式

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 单章写作、短篇（1-5章） | 常规模式 A 节 | 简单直接，无需记忆管理 |
| 连载长篇（10章以上） | **长篇写作模式 I 节** | 需要记忆系统跨 session 续写 |
| 卡文求助 | G 节（两种模式共用） | 文风分析 + 多方案引导 |
| 已有大量章节需要续写 | **长篇写作模式 I 节** | 自动读取记忆，无需手动翻找 |

## 内存文件系统详解

所有记忆文件存储于 `novel/memory/` 目录。以下是每个文件的详细说明：

### bible.md —— 核心设定（< 2000字）

**作用：** 整部小说的"宪法"，AI 每次写新章节时都会读取。

**包含内容：**
- 世界观核心规则（如"魔法消耗生命力""灵气会枯竭"）
- 力量体系/等级划分
- 不可违背的设定铁律
- 关键时间线锚点

**维护规则：**
- 严格控制字数在 2000 字以内，超出时必须精简
- 只记录"不可变"的核心设定，临时设定不写入
- 只有在作者确认新增设定时才能修改

### outline.md —— 章节大纲列表

**作用：** 整部小说的章节级索引，AI 知道当前写到哪、接下来该写什么。

**格式示例：**
```markdown
- 第1章：主角穿越，获得系统（已写）
- 第2章：第一次治疗任务，认识方明辉（已写）
- 第3章：危重病人登场，系统发布进阶任务（已写）
- 第4章：与何文涛主任的第一次冲突（正在写）
- 第5章：治疗方案分歧，主角面临选择（待写）
```

### summary.md —— 章节摘要（核心机制）

**作用：** 让 AI 快速回顾已写章节的核心事件，避免"写了后面忘前面"。

**特点：** 自动压缩、文件轮转（详见 I5 压缩算法细则）

**存储格式：**
```markdown
## ch01 - 穿越与觉醒
陈远魂穿到平行世界的外科医生身上，绑定"妙手仁心"系统。系统发布第一个任务：独立完成一台阑尾切除手术。

## ch02 - 初识搭档
陈远在手术室认识了麻醉医生方明辉。方明辉看似油嘴滑舌，实则技术过硬。两人配合完成手术。

...（每30条触发一次总纲压缩，将5章合并为一条总纲）
```

### recent.md —— 最近2章全文

**作用：** 保持写作的连贯性——对话语气、描写风格、情绪落点等摘要无法捕捉的细节。

**维护规则：**
- 只保留最后 2 章的完整正文
- 超过 6000 字时压缩为关键情节摘要
- 每写完新章节就替换：移除最早的一章，加入最新的一章

### status.md —— 当前状态

**作用：** 让 AI 时刻了解当前进度、伏笔状态和人物情绪。

**包含内容：**
```markdown
## 当前进度
- 已写章节：1-3章
- 当前章节：第4章（进行中）
- 时间线位置：陈远穿越第3天

## 未解决伏笔
- 系统的最终目的是什么？（ch01设置）
- 方明辉过去发生过什么？（ch02暗示）
- 何文涛主任似乎认识陈远的原身？（ch03结尾）

## 已回收伏笔
- （无）

## 人物情绪状态
- 陈远：对新世界感到不安，但对医学充满热情
- 方明辉：对陈远的突然转变感到好奇
- 何文涛：对陈远的态度从怀疑转为观察
```

## 会话启动流程（I1）

每次对话开始时（包括因上下文耗尽而重新开始的对话），AI 必须按顺序读取以下文件：

1. **bible.md** — 理解世界观核心设定
2. **summary.md** — 了解已写章节的情节走向
3. **recent.md** — 把握最新的写作风格和情绪落点
4. **status.md** — 明确当前进度和待解决问题
5. **相关人物卡** — 从 `novel/characters/` 读取当前出场角色的人物卡

> **注意：** 不要一次性读取所有文件全部内容。按优先级依次读取，如果上下文中已经有一些信息就跳过。

---

## 上下文管理原则（I2 / I4）

长篇写作时，以下内容**可以**留在上下文中：
- bible.md（始终需要）
- summary.md + 扩展文件（按需）
- recent.md（最多 6000 字）
- 相关人物卡

以下内容**必须**从上下文中移除：
- 超过 3 章的全文（除非当前正在处理）
- 已完成的章节原文（summary.md 代替）
- 非当前出场的人物卡

---

## 写完章节后的更新流程（I3）

每写完一章节（或每写 2000 字），立即执行以下更新。顺序不可颠倒：

**第1步：保存正文**
→ 追加到 `novel/draft/chXX.md`

**第2步：更新 summary.md**
→ 根据当前章节编号写入对应的 summary 文件
→ 第 1-130 章 → summary.md，第 131-260 → summary2.md，以此类推
→ 每新增一条时检查总条目数，达到 30 条触发总纲压缩

**第3步：更新 recent.md**
→ 移除最早的一章，加入最新的一章全文
→ 如果超出 6000 字，将最旧的一章压缩为关键情节摘要

**第4步：更新 status.md**
→ 记录新伏笔、已回收伏笔、人物情绪状态变化、当前时间线位置

**第5步：更新人物卡**
→ 检查新登场角色并创建人物卡
→ 检查已有角色的性格一致性

## 压缩算法详解（I5）

### 为什么需要压缩？

假设每章写 3000 字，100 章就是 30 万字。如果不对摘要进行压缩，summary.md 本身就会大得无法一次性读取。

### 单章压缩公式

```
压缩后字数 = max(原字数 / 10 - 100, 50)
```

| 原字数 | 压缩后 | 说明 |
|--------|--------|------|
| 500 字 | 50 字 | 短章节保底 50 字 |
| 1000 字 | 50 字 | 下限保护 |
| 2000 字 | 100 字 | 正常压缩 |
| 3000 字 | 200 字 | 标准章节 |
| 5000 字 | 400 字 | 长章节 |

### 总纲压缩流程

当 summary 文件内累积条目达到 **30 条**时触发：

```
触发前: [ch01] [ch02] [ch03] ... [ch30]  ← 30条单章摘要
              ↓ 每5章合并为一条总纲
触发后: [ch01-05总纲] [ch06-10总纲] ... [ch26-30总纲]  ← 6条总纲
```

### 文件轮转

当总纲压缩之后，文件内仍然全部是总纲（无单章条目）且条目数为 30 时：
→ 关闭当前 summary.md（不再写入新内容）
→ 创建 summary2.md，从下一章开始写入
→ summary2.md 满后 → summary3.md，以此类推

## 常见问题

**Q: 为什么不能一次性把所有章节都放入上下文？**
A: AI 上下文窗口有限。长篇写作模式下，一个典型的对话可能只能容纳：
- bible.md（~1500字）+ summary.md（~2000字）+ recent.md（~6000字）+ 人物卡（~1000字）
- 总共约 1 万字，不足 10 章的正文
- 这就是为什么必须用摘要代替全文

**Q: 对话上下文耗尽了怎么办？**
A: 只需说"继续写"，AI 会自动：
1. 重新读取所有记忆文件
2. 加载最近一章的结尾
3. 从中断处继续

**Q: summary 压缩会丢信息吗？**
A: 总纲压缩的本质是**合并**而非删除。5 章合并为一条总纲时，核心事件和转折点都会被保留。如果需要在总纲中找回细节，AI 可以根据总纲描述从 draft 目录读取对应章节的原文。

**Q: 如何切换回常规模式？**
A: 长篇写作模式和常规模式的区别在于记忆文件管理。如果想切换回常规模式，停止更新 memory/ 目录即可。已积累的记忆文件不会丢失。
